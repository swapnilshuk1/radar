# Product Constitution & Master Specification: RADAR

RADAR exists to eliminate executive career noise. Every feature must reduce uncertainty, shorten decision time, or increase decision quality. Features that do none of these should not be built.

---

## 1. Product Vision & Manifesto

RADAR is not a job board.  
RADAR is an **Executive Intelligence Platform**.

LinkedIn helps users discover jobs.  
RADAR helps executives make better career decisions.

Every screen must answer three questions:
1. **Why did RADAR surface this?**
2. **Why does it matter now?**
3. **What should I do next?**

If any UI element does not help answer one of these questions, remove it.

### The North Star Briefing
**Every opportunity in RADAR must read like a bespoke, one-page executive search consultant briefing—not like a database record or spreadsheet entry.**

---

## 2. Product Personality

- **RADAR is:** Confident, Calm, Analytical, Opinionated, Fast.
- **RADAR is NOT:** Playful, Chatty, Corporate, Salesy, Academic.

---

## 3. The Three Pillars of Evaluation

RADAR evaluates opportunities across three independent pillars:
1. **Candidate Fit:** *Can I do this job?* (Capability Fit: Title, leadership, function, industry scale) and *Do I want this job?* (Preference Fit: Location, hybrid, travel, compensation). These are kept independent and never averaged.
2. **Opportunity Quality:** *Is this a high-quality listing?* Evaluates company health, layoffs status, and financial growth.
3. **Career Trajectory Value:** *Will this job make my career stronger in five years?* Tracks title progression, P&L expansion, brand equity (employer quality), and pathways to CEO/Board readiness.

---

## 4. Core Evaluator Schema (RADAR V3.1)

Every evaluator (Title, Seniority, Functional, Location) returns an identical object interface:

```typescript
export interface MatchEvidence {
  id: string;
  label: string;
  source: "resume" | "job";
  confidence: number;
  weight: number;
}

export interface EvaluationDimension {
  score: number;
  confidence: number;
  matched: MatchEvidence[];
  missing: MatchEvidence[];
  explanation: string;
  metadata?: Record<string, unknown>;
}

export type FitVector = Record<string, EvaluationDimension>;
```

---

## 5. Ingestion Pipeline & Execution Context

The scraper parses listings and builds a shared execution context:
- **Entity Extraction Caching:** Extracted entities (Title, Seniority, Skills, Team Size) are run exactly once and cached, preventing duplicate LLM parses.
- **SkillGraphService:** Centralizes taxonomy calculations. Evaluators never handle raw taxonomy graphs directly.
- **Independent Decision Engine:** Configurable recommendation rules evaluate *only* the `FitVector` using pure code rules, preventing LLM hallucinations from affecting verdicts.
- **Evidence Provider Interface:** Agnostic AI model wrapper. The LLM consumes the pre-computed `FitVector` to output a 45-word narrative summary (no scores generated by AI).

## 6. Dimension Registry Architecture

To allow future plugins (e.g., CompanyHealth, Compensation, InterviewReadiness) to be added without changing the core engine, RADAR V3.1 exposes a static registry interface:

```typescript
export interface Evaluator {
  dimensionName: string;
  score(context: EvaluationContext): EvaluationDimension;
}

export class DimensionRegistry {
  private static evaluators = new Map<string, Evaluator>();

  public static register(evaluator: Evaluator): void {
    this.evaluators.set(evaluator.dimensionName, evaluator);
  }

  public static get(dimensionName: string): Evaluator | undefined {
    return this.evaluators.get(dimensionName);
  }

  public static executeAll(context: EvaluationContext): FitVector {
    const fitVector: FitVector = {};
    for (const [name, evaluator] of this.evaluators.entries()) {
      try {
        fitVector[name] = evaluator.score(context);
      } catch (err) {
        fitVector[name] = {
          score: 0,
          confidence: 0,
          matched: [],
          missing: [],
          explanation: `Evaluation failed: ${err}`,
        };
      }
    }
    return fitVector;
  }
}
```

## 7. RADAR Design DNA & Principles

### Emotional DNA
The console must feel closer to high-end focus products like **Linear**, **Superhuman**, **Raycast**, **Bloomberg Terminal**, and **Notion**, and avoid any resemblance to analytics or enterprise platforms like **Tableau**, **PowerBI**, **Salesforce**, **SAP**, or **Jira**.

### Enduring Principles
1. **Decisions over data.** Present clear advice, not just raw figures.
2. **Interpretation over metrics.** Raw numbers are never the end product; explanation is.
3. **Evidence over opinions.** Every recommendation must cite objective ranking engine matches and gaps.
4. **Recommendations over descriptions.** Highlight next actions instead of role summaries.
5. **Progressive disclosure over clutter.** Show decision metrics collapsed; expand for evidence; side-panel for deep context.
6. **Typography over decoration.** Use size, weight, letter-spacing, and whitespace instead of borders, cards, and outlines.
7. **Calm confidence over visual excitement.** Restrained colors only; no gradients, glassmorphism, or flashy animations.
8. **Reduce uncertainty.** Every screen must accelerate decisions and increase confidence.

---

## 8. Cognitive Load & Information Economy

- **One Glance, One Decision, One Action:** Every opportunity card should require exactly one glance to comprehend, one decision to prioritize, and one action to execute.
- **No Cross-Screen Comparison:** Never force the user to compare two separate parts of the screen to make a decision.
- **Self-Contained Context:** Never make users remember metrics from a previous row. Every card must be self-contained.
- **Information Budgets:**
  - **Collapsed Row:** Maximum 5 facts.
  - **Expanded Row:** Maximum 12 facts.
  - **Executive Brief:** Unlimited.

---

## 9. Visual & Typography System

The interface establishes visual hierarchy strictly through text style mappings:
- **Titles:** `22px` Semibold, Slate 950 (dark, prominent).
- **Companies:** `15px` Medium, Slate 700.
- **Recommendations:** `13px` Semibold, Emerald (or custom verdict color tag).
- **Metadata:** `12px` Monospaced, Slate 400.

### Eye Movement Order
The interface layout guides the eye in this exact sequence:
1. **Role Title** (Context)
2. **Action Recommendation** (Decision)
3. **Evidence / Signals** (Proof)
4. **Company Name** (Context)
5. **Metadata** (Temporal Date & Source)
6. **Triage Actions** (Controls)

---

## 10. Workspace & Grid Spacing

The viewport layout divides height according to standard console ratios:
- **Header:** 8%
- **Search:** 7%
- **Feed scroll window:** 70%
- **Footer details:** 15%

- **Collapsed row height:** Target is exactly `80px`.
- **Expanded row height:** Target is exactly `190px`.
- Whitespace exists to improve readability—not to reduce information density.

---

## 11. Interaction & Performance

- **Non-blocking Triage:** Selection never navigates away. Rows expand inline, and the Executive Brief updates immediately.
- **Hover Reveal:** Hover states reveal secondary actions only. Primary actions remain keyboard accessible at all times.
- **Subtle Motion:** Animations are subtle and brief (selection animation: `200ms`, hover states: `150ms`). Transition animations must not block or delay triage.
- **Performance Targets:** Interaction latency `<100ms`, search debounce `150ms`, zero layout shifts.

---

## 12. Design Guardrails (Negative Constraints)

- **NEVER** center content; all text and metrics must be left-aligned (or right-aligned for trailing metadata).
- **NEVER** use more than three accent colors across the entire console.
- **NEVER** use icons beside every label; icons must be reserved only for critical alerts.
- **NEVER** put labels inside boxes or thick colored pills.
- **NEVER** use more than two font weights in one component.
- **NEVER** exceed 72 characters per recommendation sentence.
- **NEVER** create a component that does not help make a decision.
- **NEVER** introduce a chart or graphic unless it directly changes the triage decision.

---

## 13. Component Behavior Specification

### Opportunity Row Component
- **States:**
  - **Idle:** Clean background, subtle left border indicator, zero border outlines.
  - **Hover (150ms transition):** Very light gray tint background (`bg-slate-50/30`), reveals secondary actions panel (Open link, Save, Archive).
  - **Selected:** Distinct left border color, light gray tint background, immediately loads details.
  - **Expanded (200ms transition):** Slides open to reveal assessment details, strengths/gaps grid, and recommended actions inline.
  - **Keyboard Focused:** Ring indicator outline matching the selection border.
  - **Loading:** Subtle font opacity transition, no heavy blocking loading overlays.

### Search Bar Component
- Minimal inline border box, clear keyboard shortcut label (`/` to focus), zero border outlines on focus (uses subtle color transitions instead).

### Score Ring Component
- Minimal circular progress stroke ring, color-coded dynamically based on the priority tier (emerald, blue, amber, slate).

### Executive Brief Panel
- Rendered using a dark neutral surface that contrasts with the light workspace while maintaining WCAG AA contrast. Scrollable independently from the feed window.

## 15. Redesign V2 Brief: Executive Brief Structure

1. **Overview Block:** Active Verdict, Fit Score, and Assessment Confidence (High/Medium/Low).
2. **Executive Summary:** Narration explaining suitability.
3. **Evidence:** Checkmarks (`✓`) showing key skills matched.
4. **Risks:** Alerts (`⚠`) showing missing requirements.
5. **Next Actions:** Single-sentence recommendation and reasoning bullet points.
6. **Intelligence Plugins (Upcoming Lock Previews):**
   - *Company Profile:* "Should I join this company?" (Preview: Size, funding, news trends, hiring velocity)
   - *Compensation Analytics:* "Is this financially attractive?" (Preview: Peer base range, bonus estimates, equity vesting)
   - *Resume & Career Optimizer:* "What should I improve to match?" (Preview: missing action verbs, tailored bullet points)
   - *Interview Readiness:* "What should I prepare?" (Preview: CEO background briefs, key discussion cards)

---

## 16. AI Communication & Copy Guidelines

RADAR summaries are written in the confident, direct voice of a retained recruiter:
- **Avoid conversational filler:** *"This role appears to be...", "Based on the description..."*
- **Prefer decisive fragments:** *"Strong commercial alignment.", "P&L ownership missing.", "Requires regional relocation."*
- **Constraints:**
  - **Summary length:** Max 45 words.
  - **Recommendations:** Exactly one sentence.
  - **No emojis, no marketing language.**

---

## 17. Product Acceptance Criteria
- [ ] UI resembles a calm, typographic executive console; no standard cards or boxes.
- [ ] Every row displays a clear actionable recommendation, not just raw database metadata.
- [ ] The Executive Brief sidebar reveals genuinely new information beyond the expanded row.
- [ ] Future intelligence plugins are pre-integrated as visual previews without breaking the panel.
- [ ] No duplicated raw text across consecutive progressive disclosure layers.
- [ ] Typography carries visual hierarchy; borders are kept to an absolute minimum.
- [ ] Color is used only to direct attention to decisions.
